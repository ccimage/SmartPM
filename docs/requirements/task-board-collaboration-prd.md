# SmartPM 任务看板与项目协作增强需求文档

**版本**：v1.0  
**日期**：2026-05-15  
**范围**：自定义看板列表、任务排序与过滤、项目成员管理、多人负责人、子任务、新建任务表单增强

---

## 一、背景与目标

当前 SmartPM 任务界面仍以固定三列 `Todo / In Progress / Done` 为核心，并且任务创建入口较轻量，只支持输入标题。随着项目协作复杂度上升，现有能力存在以下问题：

- 看板列表固定，无法按团队流程自定义阶段。
- 任务在看板中的排序和跨列编排能力不足。
- 任务过滤能力有限，难以在中大型项目中快速定位任务。
- 项目协作角色过于简单，缺少项目级权限分层和邀请管理。
- 任务负责人仅支持单人，不适合多人协作场景。
- 子任务展示和编辑链路不完整，任务拆解深度不足。
- 新建任务信息不完整，创建后仍需二次补录多个关键字段。

本轮需求目标是将当前“固定状态看板”升级为“可配置、可排序、可协作”的项目任务管理能力，并补齐项目成员管理和任务结构化编辑能力。

---

## 二、术语定义

- **看板列表**：任务看板中的一列，例如当前的 `Todo`、`In Progress`、`Done`。本次需求中，看板列表改为用户可配置对象，不再固定为 3 个预置状态。
- **列表顺序**：看板列表从左到右的排列顺序。
- **任务顺序**：同一看板列表内，任务从上到下的排列顺序。
- **项目管理员**：项目级管理角色，可管理项目成员、项目配置及任务。
- **项目成员**：参与项目协作的普通成员，可查看、创建、编辑任务。
- **负责人**：被分配到任务上的成员。本次需求支持一个任务对应多个负责人。
- **子任务**：挂载在父任务下的任务项，具有独立详情页/编辑能力。
- **回收站**：项目级删除缓冲区，用于存放被删除的任务和看板列表，支持后续恢复或彻底清理。

---

## 三、当前基线

结合现有前后端实现与设计文档，当前基线如下：

- 看板列表固定为 `todo / in_progress / done` 三种状态，由前端写死。
- 任务卡片可跨列拖动，但列表本身不可新增、删除、重命名、排序。
- 同列内任务没有显式排序字段，展示顺序依赖列表返回顺序。
- 任务过滤能力尚未形成完整产品功能。
- 项目成员接口和数据表已存在，但角色仅体现为基础成员关系。
- 任务模型已支持 `parentId`，后端已有子任务接口，但前端详情区未形成完整子任务入口。
- 任务负责人当前为单人字段（`assigneeId` / `assignee`）。
- 新建任务当前仅支持快速输入标题，更多字段需进入详情弹出框后再编辑。

## 四、详细需求

### 4.1 自定义看板列表

**需求描述**

任务看板中的列不再固定为 `Todo / In Progress / Done`，而是由项目内的“看板列表”配置驱动。

用户可在项目内新增看板列表，并为新增列表输入名称，例如：
- 待处理
- 设计中
- 开发中
- 联调中
- 已完成

每个看板列表中均可创建任务、展示任务、接收拖拽任务。

**业务规则**

- 项目初始化时，系统默认创建 3 个基础看板列表，名称可沿用当前默认值。
- 新增看板列表时，名称必填，长度建议 1–30 字。
- 同一项目内允许看板列表重名，通过系统内部唯一 `id` 区分。
- 看板列表应具备独立标识 `id`、显示名称 `name`、排序字段 `order`。
- 任务不再仅依赖固定 `status` 字段归属看板，而应关联所属看板列表 `boardListId`。
- 原有固定状态字段若保留，仅作为兼容/统计用途，不再驱动前端主看板展示。
- 看板列表支持删除，但删除前必须进行二次确认。
- 删除看板列表时，该看板列表中的任务一并删除。
- 被删除的看板列表及其任务不直接物理删除，而是进入项目回收站。
- 在交互文案中，“看板列表”和“任务清单”可以视为同一概念，面向用户展示时可优先使用更易理解的“任务清单”表达。

**验收标准**

- 项目任务页可显示项目下全部看板列表。
- 用户可以新增看板列表并立即显示在看板中。
- 新建的看板列表支持添加任务。
- 用户可以删除看板列表，删除前有二次确认。
- 删除后可在回收站中看到被删除的看板列表及任务。
- 刷新页面后，自定义看板列表仍保持正确显示。

---

### 4.2 看板列表排序

**需求描述**

看板列表支持拖动排序，用户可以调整列的先后顺序。

**业务规则**

- 拖动操作为从左到右重新排列。
- 排序结果需持久化到后端，刷新后顺序保持不变。
- 看板列表排序仅对当前项目生效，不影响其他项目。
- 排序过程中应有明确的拖拽反馈。

**验收标准**

- 任一看板列表可被拖动到其他列表前后位置。
- 拖动完成后界面顺序立即更新。
- 刷新页面后顺序与拖动后结果一致。

---

### 4.3 任务过滤

**需求描述**

任务看板页增加过滤能力，可按任务属性字段过滤任务，标题过滤支持模糊查询。

**过滤范围建议**

- 标题：模糊匹配
- 负责人：单选或多选
- 截止时间：例如已过期、今天、近 7 天、自定义时间范围
- 标签：多选
- 优先级：多选
- 看板列表：多选
- 是否父任务 / 是否子任务

**业务规则**

- 标题过滤使用模糊查询，输入关键字后过滤标题包含该关键字的任务。
- 多个过滤条件可叠加生效。
- 过滤结果仅影响当前视图展示，不改动任务实际数据。
- 过滤状态应在页面内可见，并支持一键清空。
- 过滤条件需要同步到 URL，便于刷新后保留与分享当前筛选视图。

**验收标准**

- 用户可按标题关键字筛选任务，支持模糊匹配。
- 用户可按负责人、截止时间、标签、优先级等字段筛选任务。
- 多条件筛选时结果正确。
- 刷新页面或复制当前 URL 后，过滤条件可正确恢复。
- 可快速清空全部过滤条件并恢复全量任务视图。

---

### 4.4 看板中任务排序

**需求描述**

同一看板列表中的任务支持拖动排序，不仅可以跨列表移动，也可以在同列表内调整上下顺序。

**业务规则**

- 每个任务需要具备显式排序字段，例如 `order` 或 `sortIndex`。
- 同列表拖动后更新任务顺序。
- 跨列表拖动时，同时更新任务所属看板列表和新顺序。
- 排序结果需持久化，刷新后不丢失。
- 当存在过滤条件时，排序交互的处理策略需谨慎：
  - 建议仅对当前真实顺序进行重排，而不是只对过滤结果局部重排。

**验收标准**

- 同一看板列表中任务可上下拖动排序。
- 不同看板列表之间拖动任务时，位置与归属都正确更新。
- 刷新后顺序保持一致。

---

### 4.5 项目成员管理与权限分层

**需求描述**

项目成员角色分为两类：

- 项目管理员
- 项目成员

两类角色都可以：
- 邀请成员进入项目
- 创建任务
- 编辑任务

项目管理员额外应具备更高管理权限，至少包括：
- 调整项目成员角色
- 移除项目成员
- 修改项目基础配置（如后续扩展需要）

**业务规则**

- 项目成员列表页需要展示成员、角色、邀请入口、移除入口。
- 邀请成员通过邮件发出，系统生成一次性邀请链接。
- 被邀请人点击邀请链接后进入注册流程，注册完成后自动加入对应项目。
- 项目管理员和项目成员都能邀请成员，是本次显式需求；因此邀请权限不能仅限管理员。
- 角色变更仅允许项目管理员执行。
- 移除成员仅允许项目管理员执行。
- 至少保留一个项目管理员，防止项目失管。

**验收标准**

- 项目内可查看成员列表和每个成员的角色。
- 可邀请新成员加入项目。
- 可区分“项目管理员”和“项目成员”。
- 两类角色都能创建和编辑任务。
- 项目管理员可调整成员角色、移除成员。
- 邮件邀请链接仅可使用一次，过期或已使用后不可再次用于加入项目。

---

### 4.6 多负责人

**需求描述**

任务负责人支持多选，一个任务可分配给多个负责人。

**业务规则**

- 负责人选择范围应限制为项目成员。
- 任务卡片和任务详情都应展示多名负责人。
- 负责人为空时允许保存，表示暂未分配。
- 新建任务与编辑任务都支持设置多个负责人。
- 原有单人负责人字段需要升级为多人关系结构，例如 `task_assignees`。
- 主任务详情中的负责人展示规则为“头像 + 姓名”，最多展示 3 个，超过后显示 `...`。
- 看板列表中的任务卡片负责人仅显示头像，hover / 气泡提示姓名，超过 2 个后显示 `...`。
- 子任务列表中的负责人仅显示头像，hover / 气泡提示姓名，超过 2 个后显示 `...`。

**验收标准**

- 用户可在任务中选择多个负责人。
- 任务卡片和详情页均能展示多个负责人。
- 负责人在主任务详情、看板卡片、子任务列表中的展示方式符合上述规则。
- 保存后刷新页面，多负责人信息仍正确。

---

### 4.7 子任务增强

**需求描述**

任务支持添加子任务，并在父任务详情中展示子任务列表。子任务至少展示：

- 子任务标题
- 负责人
- 截止日期

点击子任务标题后，进入该子任务的详细编辑界面。

**业务规则**

- 子任务为独立任务实体，继承当前项目上下文。
- 子任务可以有自己的负责人、截止时间、标签、描述等字段。
- 父任务详情内展示的子任务为列表形式。
- 子任务列表中，标题、负责人、截止日期三项支持内嵌快速编辑。
- 点击子任务标题后，进入同一套任务详情弹出框/编辑界面，加载对应子任务详情。
- 子任务层级本期建议限制为一级，即“父任务 + 子任务”，不支持孙任务。

**验收标准**

- 父任务中可新增子任务。
- 子任务列表展示标题、负责人、截止日期。
- 子任务列表支持对标题、负责人、截止日期进行内嵌快速编辑。
- 点击子任务标题可进入子任务详情编辑界面。
- 子任务保存后，父任务中的子任务列表即时更新。

---

### 4.8 新建任务表单增强

**需求描述**

当前新建任务仅能输入标题。需要将新建任务能力增强为完整表单，增加以下字段：

- 负责人
- 截止时间
- 描述

结合已有字段，任务创建应分为两种模式：

**快速新增**：
- 标题（必填）
- 负责人（可选）
- 描述（可选）
- 截止时间（可选）

**完整编辑框**：
- 标题
- 看板列表归属
- 优先级
- 负责人（多选）
- 截止时间
- 标签
- 描述

**业务规则**

- 标题必填，其他字段可按业务决定是否必填；本期建议除标题外均可选。
- 快速新增和完整编辑框并存。
- 快速新增用于低成本录入任务，字段少于完整编辑框。
- 标签不进入快速新增，仅在完整编辑框中填写。
- 描述字段应继续使用现有富文本编辑器。
- 快速新增成功后，任务应直接出现在对应看板列表中。
- 完整编辑框用于补充更完整任务信息。

**验收标准**

- 快速新增界面可输入标题、负责人、描述、截止时间。
- 完整编辑框可输入标题、看板列表、优先级、负责人、截止时间、标签、描述。
- 保存后任务卡片正确展示相关信息。
- 新建任务后再次打开详情，字段值与创建时一致。

## 五、数据与接口影响

本需求会显著影响现有数据模型和接口设计，至少包括以下内容：

### 5.1 数据模型建议调整

- 新增 `board_lists`（或等价表）
  - `id`
  - `project_id`
  - `name`
  - `order`
  - `deleted_at`（如采用软删除）
  - `created_at`
  - `updated_at`

- 任务表 `tasks`
  - 增加 `board_list_id`
  - 增加排序字段 `order`
  - 保留 `parent_id`
  - 增加 `deleted_at` / 回收站相关标记（如采用软删除）
  - 原单人负责人字段需评估是否废弃或兼容

- 新增任务负责人关联表，例如 `task_assignees`
  - `task_id`
  - `user_id`

- 项目成员表 `project_members`
  - `role` 明确支持 `admin` / `member`

- 新增项目邀请或邀请令牌相关表，例如 `project_invites`
  - `id`
  - `project_id`
  - `email`
  - `token`
  - `role`
  - `invited_by`
  - `expires_at`
  - `used_at`

### 5.2 接口能力建议补充

- 看板列表 CRUD 与排序接口
- 看板列表删除与回收站接口
- 任务排序接口
- 任务多负责人读写接口
- 项目成员角色变更接口
- 项目邮件邀请接口与邀请注册链接校验接口
- 任务列表高级过滤查询参数
- 新建/编辑任务接口支持多负责人、标签、描述、截止时间完整提交

## 六、交互与页面建议

### 6.1 看板页

- 顶部增加过滤栏
- 看板列表头部支持重命名/更多操作
- 每列底部保留“新增任务”入口
- 看板页提供回收站入口
- 快速新增任务表单默认折叠，展开后提供标题、负责人、描述、截止时间四项
- 任务卡片展示：
  - 标题
  - 多负责人头像摘要（hover 显示姓名）
  - 截止时间
  - 标签
  - 子任务进度摘要（可选）

### 6.2 任务详情弹出框

- 保持现有弹出框编辑形态
- 增加多负责人字段
- 增加子任务区域
- 子任务区域支持标题、负责人、截止日期三项内嵌编辑
- 子任务点击标题后切换到对应任务详情

### 6.3 项目成员管理页

- 建议作为项目内独立页面或弹出层
- 展示成员头像、姓名、角色、加入时间（可选）
- 支持输入邮箱邀请成员
- 支持角色修改、移除

### 6.4 回收站

- 项目级提供回收站页面或弹出层
- 展示已删除的看板列表与任务
- 支持恢复和彻底删除
- 删除看板列表时，相关任务应与看板列表一起进入回收站

---

## 七、非本次范围

- 工作区级更细粒度权限体系
- 子任务三级及以上层级
- 甘特图、日历视图
- 自动化规则引擎（如进入某列自动触发通知）
- 看板列表颜色/图标配置

---

## 八、成功标准

- 项目看板从固定三列升级为可配置看板列表，适应不同团队流程。
- 看板列表和任务均支持拖动排序，并可持久化保存。
- 任务过滤能力覆盖标题模糊查询和核心属性筛选。
- 项目成员具备项目管理员 / 项目成员双角色模型，协作权限更清晰。
- 邮件邀请、一次性注册链接和项目加入流程可用。
- 看板列表与任务删除后进入回收站，可恢复或彻底删除。
- 任务支持多负责人和子任务，任务拆解能力更完整。
- 新建任务同时支持快速新增和完整编辑两种模式，兼顾效率与完整性。
